# Lean Backend: Proving Properties of Specifications

**Table of Contents**

---

In order to reason about the contracts and prove their properties, the Lean backend of act shallowly
embeds the act specification into the logic of the Lean 4 proof assistant, using Mathlib as the
underlying mathematical library.

For this embedding to be sound, we ensure that the storage of the
contract (and the contracts reachable from it) satisfies the *unique ownership invariant* (see [aliasing section](./aliasing.md)), meaning that its storage may never contain any aliased references to other contracts.
act checks this invariant automatically immediately after the type-checking phase.

## Usage

**Generate Lean formalization:**
To generate the Lean code run
```sh
act lean --file <PATH_TO_SPEC>
```
against your spec, or
```sh
act lean --json <PATH_TO_JSON>
```
where the JSON file has the following format
```json
{
  "specifications" : ["<PATH_TO_SPEC>",..],
}
```

Note: In the current implementation the specifications must be ordered, so that any specification only depends on others earlier in the list.

#### Lean Command-Line Flags

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--file` | Path | - | Path to the act specification file (.act) |
| `--json` | Path | - | Path to the sources JSON file (.json) |
| `--solver` | `cvc5\|z3` | `cvc5` | SMT solver used during type-checking and unique ownership verification |
| `--smttimeout` | Integer (ms) | `20000` | Timeout for SMT queries |
| `--debug` | Boolean | `false` | Print verbose output during generation |

### ActLib

The generated Lean code imports `ActLib`, which provides the foundational definitions for reasoning
about EVM contracts (the `Env` type, address bounds, `multistep`, and helper lemmas). The `ActLib`
source lives in the `lib/` directory of the act repository and is a Lake package that depends on
Mathlib. To use it in your own Lean project, add the following to your `lakefile.toml`:

```toml
[[require]]
name = "ActLib"
path = "/path/to/act/lib"
```

and add a `require mathlib` entry if you do not already have one.

Make sure to run `lake update` in the `lib/` directory and also in your project directory after modifying your `lakefile.toml` to pull in the new dependency.

## act Export

Calling `act lean ...` will generate a Lean file that encodes the contract as a state transition
system, following the formal value semantics given in the <span style="color:green">tech report (to be available shortly).</span>

The generated file begins with imports and opens the relevant namespaces:

```lean
-- GENERATED BY ACT

import Mathlib.Data.Int.Basic
import ActLib

open Int
open ActLib
```

Each contract in the specification is placed in its own `namespace`. The generated output contains:

- A **`State` structure** with fields for every storage variable, plus an `addr` field for the
  contract's own address.
- **`addressIn`** and **`noAliasing`** inductive predicates.
- **`stateIntegerBounds`** — an inductive predicate asserting all storage variables are within
  their declared integer ranges.
- **`nextAddrConstraint`** — a definition asserting that `NextAddr` is greater than every address
  currently stored in the state.
- **`envNextAddrConsistency`** and **`stateConsistency`** — inductive predicates capturing
  well-formedness conditions on the execution environment.
- For the constructor and every transition:
  - A **preconditions predicate** (e.g. `constructor_conds`, `transfer_conds`).
  - A **state transition relation** (e.g. `constructor`, `transfer_transition`).
  - A **return value predicate** (e.g. `transfer_ret`) — only for transitions that specify a return value.
- A **`Token_transition`** relation collecting all the contract's own transitions.
- A **`transition`** relation extending `Token_transition` with transitions of any sub-contracts
  stored in the state.
- **`step`**, **`multistep`**, **`reachable`**, and **`reachableFromInit`** — the standard
  reachability definitions for the transition system.
- **`stepBefore`**, **`multistepBefore`**, and **`reachableBefore`** — bounded variants used when
  reasoning about sub-contract reachability with respect to an address bound.
- Any **postcondition** definitions declared with `ensures` in the specification.
- A parametric **invariant schema**: `invariantInit`, `invariantStep`, and a proved
  `invariantReachable` theorem.

### Example: ERC20 Token

Let us explore the generated Lean output for the ERC20 Token contract from [erc20.act](https://github.com/argotorg/act/blob/v0.2.0_documentation/tests/lean/ERC20/erc20.act).

The contract state is represented as a Lean `structure`:

```lean
namespace Token

structure State where
  addr : address
  BALANCE : ℤ
  allowance : address → address → ℤ
  balanceOf : address → ℤ
  totalSupply : ℤ
```

The fields correspond to the storage variables declared in the act specification. The `addr` field
holds the contract's own address, and `BALANCE` holds the amount of wei sent to the contract.
Integers are represented as `ℤ` (Lean's arbitrary-precision integers from Mathlib), and `address`
is defined in `ActLib` as an alias for `ℤ`.

The `ActLib` provides an **`Env`** structure representing the execution environment:

```lean
structure Env where
  Callvalue : ℤ
  Caller : address
  Origin : address
```

`NextAddr` is a parameter (of type `address`) threaded through constructor and transition relations.
It represents the next address to be allocated by the EVM, and appears as a pair of values (before
and after the call) in each relation.

The **preconditions for the constructor** are generated as an inductive predicate:

```lean
inductive constructor_conds (ENV : Env) (_totalSupply : ℤ) (NextAddr : address) : Prop where
| constructor_condsC : ∀
  ( H_iff0 : (ENV.Callvalue) = 0 )
  ( H_argConstraints_intBounds__totalSupply : 0 ≤ _totalSupply ∧ _totalSupply ≤ UINT_MAX 256 ),
  constructor_conds ENV _totalSupply NextAddr
```

The hypothesis `H_iff0` reflects the `iff` block in the act specification (the constructor is not
payable). The remaining hypotheses enforce integer bounds on the arguments.

The **constructor state transition relation** has the following form:

```lean
inductive constructor (ENV : Env) (_totalSupply : ℤ) (NextAddr : address)
    : State → address → Prop where
| Token_case0 : ∀ NextAddr1
  ( H_conds : constructor_conds ENV _totalSupply NextAddr )
  ( H_case_cond : True )
  ( H_envNextAddrConsistent : envNextAddrConsistency ENV NextAddr )
  ( H_bindings0 : NextAddr1 = NextAddr + 1 ),
  constructor ENV _totalSupply NextAddr
    (State.mk NextAddr 0
      (fun _ => fun _ => 0)
      (fun a => if a == ENV.Caller then _totalSupply else 0)
      _totalSupply)
    NextAddr1
```

The constructor relates the environment, input arguments, and the next address, to a new state and
the updated next address. The contract is stored at `NextAddr`. The `balanceOf` mapping initialises
the caller's balance to the total supply.

Similarly, **preconditions and transition relations are generated for every transition**. For the
`transfer` transition:

```lean
inductive transfer_conds (ENV : Env) (_value : ℤ) (_to : address)
    (STATE : State) (NextAddr : address) : Prop where
| transfer_condsC : ∀
  ( H_iff0 : (ENV.Callvalue) = 0 )
  ( H_iff1 : ... )  -- caller has sufficient balance and value is in range
  ( H_iff2 : ... )  -- recipient balance does not overflow (when CALLER ≠ _to)
  ( H_nextAddrCnstrnt_State : nextAddrConstraint NextAddr STATE )
  ( H_argConstraints_intBounds__value : 0 ≤ _value ∧ _value ≤ UINT_MAX 256 )
  ( H_argConstraints_intBounds__to : 0 ≤ _to ∧ _to ≤ UINT_MAX 160 ),
  transfer_conds ENV _value _to STATE NextAddr
```

The **return value predicate** for `transfer` is:

```lean
inductive transfer_ret (ENV : Env) (_value : ℤ) (_to : address)
    (STATE : State) (NextAddr : address) : Bool → Prop where
| transfer_case0_ret :
     transfer_conds ENV _value _to STATE NextAddr
  → (ENV.Caller) ≠ _to
  → transfer_ret ENV _value _to STATE NextAddr true
| transfer_case1_ret :
     transfer_conds ENV _value _to STATE NextAddr
  → (ENV.Caller) = _to
  → transfer_ret ENV _value _to STATE NextAddr true
```

The **step and reachability definitions** are:

```lean
def step (STATE STATE' : State) :=
  ∃ (ENV : Env) (NextAddr NextAddr' : address),
    transition ENV STATE NextAddr STATE' NextAddr'

def multistep (STATE STATE' : State) :=
  ActLib.multistep step STATE STATE'

def reachable (STATE : State) :=
  ∃ STATE', init STATE' ∧ multistep STATE' STATE

def reachableFromInit (ENV : Env) (_totalSupply : ℤ) (NextAddr : address)
    (STATE : State) :=
  ∃ STATE' NextAddr',
    constructor ENV _totalSupply NextAddr STATE' NextAddr'
    ∧ multistep STATE' STATE
```

`ActLib.multistep` is the reflexive transitive closure of a binary relation, defined in `ActLib`
using `Relation.ReflTransGen` from Mathlib.

Finally, the generated file contains a parametric **invariant schema**. Given an invariant property
`IP : Env → ℤ → address → State → Prop`, the following are generated:

```lean
def invariantInit (IP : Env → ℤ → address → State → Prop) :=
  ∀ (ENV : Env) (_totalSupply : ℤ) (NextAddr : address)
    (STATE : State) (NextAddr' : address),
      constructor ENV _totalSupply NextAddr STATE NextAddr'
    → IP ENV _totalSupply NextAddr STATE

def invariantStep (IP : Env → ℤ → address → State → Prop) :=
  ∀ (ENV : Env) (_totalSupply : ℤ) (NextAddr : address)
    (STATE STATE' : State),
      constructor_conds ENV _totalSupply NextAddr
    → step STATE STATE'
    → IP ENV _totalSupply NextAddr STATE
    → IP ENV _totalSupply NextAddr STATE'
```

Along with a **proved theorem** `invariantReachable`:

```lean
theorem invariantReachable :
  ∀ (ENV : Env) (_totalSupply : ℤ) (NextAddr : address) (STATE : State)
    (IP : Env → ℤ → address → State → Prop)
    (HIPinvInit : invariantInit IP)
    (HIPinvStep : invariantStep IP),
      reachableFromInit ENV _totalSupply NextAddr STATE
    → IP ENV _totalSupply NextAddr STATE
```

The proof is included in the generated file and uses `ActLib.step_multi_step`.

## Proving properties using the act Lean export

Once the Lean file is generated, you can import it from a separate file to write your own proofs.
This is the recommended workflow, so that the generated file can be regenerated without affecting
your manual work.

```lean
import MyContract  -- the generated file
import ActLib

open Token  -- the namespace for the contract

-- Prove invariants and properties
theorem my_invariant : ∀ STATE, reachable STATE → ... := by
  ...
```

To use the `invariantReachable` theorem, define your invariant property `IP`, prove `invariantInit IP`
and `invariantStep IP`, and apply:

```lean
theorem my_invariant_reachable :
    ∀ ENV _totalSupply NextAddr STATE,
    reachableFromInit ENV _totalSupply NextAddr STATE →
    IP ENV _totalSupply NextAddr STATE :=
  invariantReachable HIPinvInit HIPinvStep
```
